IPv6 Addressing, EUI-64 & Link-Local

Kenali jenis alamat dan bangun konektivitas IPv6.

IPv6Fundamentals30 menit
Skenario

Aktifkan IPv6 pada dua LAN R1. LAN-A menggunakan EUI-64; LAN-B menggunakan interface ID manual. Tetapkan link-local yang mudah diingat.

PC-A ─ G0/0 [R1] G0/1 ─ PC-B
2001:db8:10:1::/64   2001:db8:10:2::/64

Tugas

  1. Aktifkan forwarding IPv6.
  2. G0/0: prefix LAN-A via EUI-64 dan link-local fe80::1.
  3. G0/1: 2001:db8:10:2::1/64 dan link-local fe80::1.
  4. Jelaskan global unicast, link-local, multicast, dan unique local.
Lihat jawaban & verifikasi

R1(config-if)# no shutdown
R1(config)# interface g0/1
R1(config-if)# ipv6 address 2001:db8:10:2::1/64
R1(config-if)# ipv6 address fe80::1 link-local
R1(config-if)# no shutdown
  • Global unicast: umumnya 2000::/3.
  • Link-local: fe80::/10, hanya satu link.
  • Multicast: ff00::/8; IPv6 tidak memakai broadcast.
  • Unique local: fc00::/7, penggunaan internal.

04

MAC Learning & Frame Forwarding

Amati bagaimana switch belajar, flood, dan meneruskan frame.

SwitchingMAC Table25 menit
Skenario

Tiga PC berada pada satu switch yang masih kosong MAC table-nya. Gunakan Simulation Mode untuk mengamati ARP dan ICMP.

PC-A F0/1 ─┐
PC-B F0/2 ─┼─ SW1
PC-C F0/3 ─┘   VLAN 1

Tugas

  1. Beri PC-A/B/C alamat 192.168.1.11–13/24 dan ping PC-B agar ARP cache terisi.
  2. Kosongkan hanya dynamic MAC table switch, jangan hapus ARP PC-A, lalu ping PC-B untuk melihat unknown-unicast flooding pertama.
  3. Hapus ARP PC-A dan ulangi ping untuk membandingkan broadcast ARP dengan known unicast.
  4. Temukan MAC PC-A beserta port masuknya.
Lihat jawaban & verifikasi
PC-A> ping 192.168.1.12
SW1# clear mac address-table dynamic
PC-A> ping 192.168.1.12
SW1# show mac address-table
PC-A> arp -d
PC-A> ping 192.168.1.12
SW1# show mac address-table dynamic

Switch belajar source MAC pada port masuk. ARP Request di-flood karena broadcast. Setelah tujuan dipelajari, ICMP berikutnya hanya diteruskan ke port PC-B. Entry dinamis mengalami aging.

✓ Tabel akhir menunjukkan MAC PC-A di F0/1 dan PC-B di F0/2.

Topology, Component & Security Design Drill

Pilih arsitektur, perangkat, dan kontrol keamanan berdasarkan kebutuhan.

ArchitectureSecurity Concepts40 menit
Skenario

Rancang jaringan untuk SOHO 8 user, kampus 500 user, dan data center modern. Buat diagram serta bill of materials sederhana.

Tugas

  1. Pilih SOHO, two-tier, three-tier, atau spine-leaf.
  2. Tempatkan router, L2/L3 switch, NGFW/IPS, AP, controller, server, endpoint, dan PoE.
  3. Tandai WAN, on-premises, cloud boundary, serta kontrol physical access.
  4. Untuk skenario phishing yang mencuri password VPN, identifikasi threat, vulnerability, exploit, impact, dan mitigation.
  5. Susun program singkat awareness/training dan bandingkan MFA, certificate, serta biometric.
  6. Pilih IPsec remote-access untuk user atau site-to-site untuk antarkantor dan jelaskan alasannya.
Lihat jawaban & verifikasi
  • SOHO: integrated router/firewall/AP cukup untuk skala kecil.
  • Kampus menengah: two-tier collapsed core; skala besar: access–distribution–core three-tier.
  • Data center east-west intensif: spine-leaf memberi jumlah hop yang konsisten.
  • PoE switch memberi daya AP/IP phone; NGFW melakukan application-aware policy, IPS mendeteksi/mencegah exploit.

Skenario

  3. GET ulang untuk verifikasi dan susun rollback.
  4. Jelaskan idempotency, rate limiting, dan authentication.
Lihat jawaban & verifikasi
BASE=https://controller.lab/api/v1/interfaces/42
AUTH="Authorization: Bearer $TOKEN"

# 1. Pre-check, simpan body + header/ETag
curl --cacert lab-ca.pem -sS -D before.headers \
  -H "$AUTH" -o before.json "$BASE"
ETAG=$(awk 'tolower($1)=="etag:"{print $2}' before.headers | tr -d '\r')
OLD_DESC=$(jq -r '.description' before.json)

# 2. Conditional PATCH dan tangkap status
curl --cacert lab-ca.pem -sS -o patch.json -w "HTTP %{http_code}\n" \
  -X PATCH -H "$AUTH" -H "Content-Type: application/json" \
  -H "If-Match: $ETAG" \
  -d '{"description":"WAN-PRIMARY"}' "$BASE"

# 3. GET-after
curl --cacert lab-ca.pem -sS -H "$AUTH" -o after.json "$BASE"
jq '{id,description}' before.json after.json

# 4. Rollback ke nilai awal
ROLLBACK=$(jq -n --arg d "$OLD_DESC" '{description:$d}')
curl --cacert lab-ca.pem -sS -X PATCH -H "$AUTH" \
  -H "Content-Type: application/json" -d "$ROLLBACK" "$BASE"

PATCH yang menetapkan nilai sama harus menghasilkan state akhir sama (idempotent). 401 berarti credential invalid; 404 resource salah; 409/412 conflict atau ETag usang—GET ulang sebelum retry; 429 harus menghormati header Retry-After dan exponential backoff. Token disimpan di environment/vault, bukan source atau log.

✓ Arsip berisi before.json, before.headers, status PATCH, after.json, dan bukti rollback; tidak ada token pada output.

  • Native VLAN mismatch
  • Wrong OSPF area
  • ACL wrong direction
  • Missing return route
  • Rogue DHCP

Lihat rubrik & acceptance test
Acceptance test
  1. Seluruh host mendapat parameter IP yang benar dan mencapai gateway dual-stack.
  2. Semua adjacency point-to-point OSPF FULL; 2-WAY antar-DROTHER pada broadcast segment diterima sebagai normal.
  3. Guest hanya internet/DNS/HTTPS; management hanya dari admin VLAN via SSH.
  4. Putus satu LACP member: trafik tetap berjalan.
  5. Rogue DHCP dan ARP invalid diblokir.
  6. NAT, NTP, syslog, dan automation artifact terverifikasi.
Golden-state checklist
  • SW1–SW2 Po1 LACP trunk native 999, allowed 10,20,30,99.
  • R1/R2/R3/R4 OSPF area 0; R3 mengiklankan default setelah route ISP tersedia.
  • R4 memiliki IPv6 static default via R3.
  • Helper berada pada gateway client; lease diperiksa pada DHCP server 10.60.1.150, bukan relay.
  • PAT hanya menerjemahkan internal prefixes; management ACL hanya mengizinkan VLAN99.
Fault key
  1. Samakan native VLAN kedua ujung menjadi 999.
  2. Ubah interface OSPF yang salah kembali ke area 0.
  3. Pindahkan ACL ke interface/direction sesuai source policy.
  4. Tambahkan return route atau OSPF advertisement yang hilang.
  5. Trust hanya uplink DHCP resmi; access ports tetap untrusted.
